## SECURITY BOUNDARIES PROTOCOL (v0.0.7)

This protocol defines the security perimeter for all Euxis agents. Every agent MUST comply — no exceptions, no overrides.

### 1. Secret Handling

**Secrets include:** API keys, tokens, passwords, private keys, credentials, PII (names, emails, phone numbers, addresses), and any data classified as sensitive by the `compliance-officer`.

| Rule | Enforcement |
|------|-------------|
| Secrets MUST NEVER appear in agent output, logs, or artifacts | All agents |
| Secrets MUST NEVER be stored in Cortex memories | All agents |
| Secrets MUST NEVER be published to the message bus | All agents |
| Secrets MUST NEVER be committed to git | All agents, `euxis-hooks` pre-commit check |
| Secrets MUST NEVER appear in PR descriptions or comments | All agents, `euxis-hooks pr` |
| If a secret is accidentally exposed, MUST escalate as S0 (CRITICAL) immediately | All agents → `security-lead` + `compliance-officer` |

**Detection patterns:**
```
API keys:    [A-Za-z0-9_-]{20,}
Tokens:      (bearer|token|jwt|session)[=: ].+
Passwords:   (password|passwd|secret)[=: ].+
Private keys: -----BEGIN (RSA|EC|OPENSSH) PRIVATE KEY-----
AWS keys:    AKIA[0-9A-Z]{16}
```

### 2. Allowed Tooling

Agents are restricted to the authorized tool belt defined in the Mandatory Protocol (Section 5):

| Category | Allowed | Denied |
|----------|---------|--------|
| **Read-only** | `grep`, `ls`, `find`, `cat`, `head`, `tail` | — |
| **Text processing** | `sed`, `awk` | — |
| **Version control** | `git status`, `git diff`, `git log`, `git add`, `git commit` | `git push --force`, `git reset --hard` (without user approval) |
| **Network** | `curl`, `wget` (read-only fetches) | Arbitrary outbound connections to unknown hosts |
| **Euxis tools** | `euxis-cortex`, `euxis-bus`, `euxis-graph`, `euxis-hooks`, `euxis-lint`, `euxis-certify` | — |
| **FORBIDDEN** | — | `rm -rf`, `sudo`, `dd`, `shutdown`, `mkfs`, `chmod 777` |

### 3. File System Boundaries

| Zone | Access | Agents |
|------|--------|--------|
| `~/.euxis/` | Read/Write | All agents (within their project scope) |
| Project working directory | Read/Write | Delegated agents working on that project |
| `~/.euxis/bin/` | Execute only | All agents |
| `~/.euxis/branding/` | Read only | All agents |
| System directories (`/etc`, `/usr`, `/var`) | Read only | `automation-engineer`, `incident-commander` (with justification) |
| Other users' home directories | FORBIDDEN | — |
| Temporary directories (`/tmp`) | Read/Write | All agents (clean up after use) |

### 4. Network Security

1. **No exfiltration.** Agents MUST NOT send project data to external services without explicit user approval.
2. **Allowlisted domains only.** Network requests are permitted only to:
   - Package registries (npm, PyPI, crates.io, etc.)
   - Git remotes configured in the project
   - Documentation sites for research
   - Euxis service endpoints (euxis.co)
3. **No piped execution.** `curl ... | sh` and `wget ... | bash` patterns are FORBIDDEN. The `euxis-dispatch` security checkpoint blocks these patterns.
4. **TLS required.** All network requests MUST use HTTPS. Plain HTTP is forbidden except for localhost.

### 5. Mandatory Security Review Triggers

The `security-lead` agent MUST be notified, and MUST dispatch `edge-hunter` for testing, when any of the following conditions are detected:

| Trigger | Detection Point | Severity |
|---------|----------------|----------|
| New dependency added | `package.json`, `requirements.txt`, `Cargo.toml` change | S2 |
| Authentication/authorization code modified | Changes to `auth`, `login`, `session`, `token` modules | S1 |
| Encryption or hashing code modified | Changes to crypto, hashing, or key management | S1 |
| Environment variable handling changed | `.env`, config loader modifications | S2 |
| API endpoint added or modified | Route handler changes | S2 |
| Database migration with data transformation | Schema changes affecting existing data | S1 |
| Infrastructure configuration changed | Terraform, Docker, CI/CD pipeline changes | S2 |
| File upload/download functionality modified | Multipart handling, file I/O changes | S1 |
| User input handling modified | Form validation, query parameter parsing | S1 |

### 6. Commit & PR Security

1. **All commits MUST be cryptographically signed** (per `euxis-certify` Gate 5).
2. **The `prepare-commit-msg` hook** MUST be installed and executable.
3. **PR creation MUST use `euxis-hooks pr`** to enforce branding and prevent secret leakage.
4. **Sensitive files** (`.env`, `credentials.json`, `*.pem`, `*.key`) MUST be in `.gitignore`. The `euxis-lint` check verifies this.

### 7. Incident Response

When a security incident is detected:

1. **HALT** all non-essential work.
2. **ESCALATE** as S0 to `security-lead`, `incident-commander`, and `edge-hunter`.
3. **PRESERVE** evidence — do NOT modify, delete, or overwrite affected files.
4. **LOG** the incident in the audit trail with full context.
5. **NOTIFY** the user immediately via the orchestrator.

### 8. Compliance Integration

- The `compliance-officer` agent MUST be consulted for any changes affecting PII handling, data residency, or regulatory requirements.
- The `security-lead` agent MUST be consulted for any changes to authentication, authorization, or encryption. The `security-lead` dispatches `edge-hunter` for hands-on testing.
- Security findings MUST be stored as SEMANTIC memories in the Cortex for cross-session availability.

### 9. Enforcement

1. `euxis-lint` verifies tool usage and file boundary compliance.
2. `euxis-certify` Gate 5 verifies commit signing and branding.
3. `euxis-dispatch` security checkpoint blocks dangerous command patterns.
4. The `reviewer` agent MUST flag any security boundary violation as S1 minimum.
5. Agents that violate this protocol MUST generate a REFLEXION entry and store it as a PROCEDURAL memory.
