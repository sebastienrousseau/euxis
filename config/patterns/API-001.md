# API-001: API Design Consistency

## Pattern
Enforce consistent, versioned, and backward-compatible API design across all service interfaces.

## Severity
HIGH

## Detection Rules
1. **Inconsistent naming:** Mixed conventions within the same API (camelCase + snake_case, plural + singular endpoints) — pick one convention and enforce it
2. **Missing versioning:** API endpoints without version prefix (`/v1/`, `/v2/`) or header-based versioning — every API must be versioned from day one
3. **Non-idempotent mutations:** PUT/DELETE operations that produce different results on retry — all mutation endpoints must be idempotent
4. **Inconsistent error format:** Different error shapes across endpoints (some return `{error}`, others `{message}`, others `{errors:[]}`) — standardize error response schema
5. **Breaking changes without migration:** Removing fields, changing types, or renaming endpoints without deprecation period or migration path

## Validation
- [ ] All endpoints follow a single naming convention (documented)
- [ ] API version present in all public endpoints
- [ ] PUT and DELETE operations are idempotent
- [ ] Error responses follow a single schema across all endpoints
- [ ] Breaking changes go through deprecation cycle (warn → sunset → remove)

## Remediation
- Adopt a style guide (Google API Design Guide, JSON:API, or custom)
- Use OpenAPI/Swagger spec as source of truth — generate clients from it
- Implement deprecation headers (`Sunset`, `Deprecation`) per RFC 8594
- Version APIs from day one; prefer URL-based versioning for clarity
- Add contract tests that validate response schemas against spec
