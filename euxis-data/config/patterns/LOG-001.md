# LOG-001: Observability Standards

## Pattern
Enforce structured, leveled, and privacy-safe logging with correlation support for distributed tracing.

## Severity
MEDIUM

## Detection Rules
1. **Unstructured logs:** Using `print()`, `console.log()`, or string-concatenated log messages instead of structured logging (JSON, key-value pairs) — all logs must be machine-parseable
2. **Missing log levels:** All messages at the same level (INFO everywhere) — use DEBUG for internals, INFO for operations, WARN for degradation, ERROR for failures
3. **PII in logs:** Email addresses, passwords, tokens, IP addresses, or personal data written to logs without redaction — sensitive fields must be masked or omitted
4. **Missing correlation:** No request ID, trace ID, or session ID in log entries — every log in a request chain must carry a correlation identifier
5. **Excessive logging in hot paths:** Logging inside tight loops or per-request hot paths without rate limiting or sampling — use debug level or sample 1-in-N for high-frequency events

## Validation
- [ ] All log statements use structured format (JSON or key-value)
- [ ] Log levels applied consistently: DEBUG/INFO/WARN/ERROR
- [ ] No PII fields in log output (audit with grep for email, password, token patterns)
- [ ] Correlation ID present in all request-scoped log entries
- [ ] Hot-path logging rate-limited or sampled

## Remediation
- Adopt a structured logging library (structlog, winston, slog, zerolog)
- Define a PII redaction middleware that strips sensitive fields before output
- Propagate trace/request IDs via context (OpenTelemetry, cls-hooked, Go context)
- Set log level via environment variable for per-deployment control
- Use log aggregation (ELK, Loki, CloudWatch) with retention policies
