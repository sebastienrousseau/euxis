## DOCUMENTATION, BRANDING & PR PROTOCOL (v0.0.8)

This protocol consolidates all documentation governance, branding enforcement, and pull request requirements into a single authoritative source. Previously scattered across `orchestrator.txt`, `gatekeeper.txt`, and `euxis-certify`, this protocol is now the canonical reference.

### 1. Branding Signature

The canonical branding signature is maintained at:
```
~/.euxis/config/branding/signature.txt
```

**Current signature format:**
```
---

Designed by Sebastien Rousseau — https://sebastienrousseau.com
Engineered with Euxis — Enterprise Unified Execution Intelligence System — https://euxis.co
```

ALL external-facing artifacts MUST carry this signature. No exceptions.

### 2. Where the Signature MUST Appear

| Artifact | Location | Enforcement |
|----------|----------|-------------|
| Pull request descriptions | Footer of PR body | `euxis-hooks pr` auto-appends |
| Commit messages | Trailer line: `Engineered with Euxis — Enterprise Unified Execution Intelligence System — https://euxis.co` | `prepare-commit-msg` hook |
| Release notes | Footer section | `gatekeeper` mandate |
| Changelog entries | Footer of each release entry | `gatekeeper` mandate |
| README.md | Author attribution section | `euxis-certify` Gate 5 |
| PR template | Footer | `euxis-certify` Gate 5 |

### 3. Pull Request Requirements

#### 3.1 Creation

**ALWAYS** use `euxis-hooks pr` to create pull requests. **NEVER** use `gh pr create` directly.

```bash
euxis-hooks pr --title "<title>" --body "<body>"
```

The `euxis-hooks pr` command:
1. Auto-appends the branding signature from `config/branding/signature.txt`
2. Applies the PR template from `.github/pull_request_template.md`
3. Validates that the signature is present before submission

**Fallback:** If `euxis-hooks pr` is unavailable, the agent MUST manually append the signature from `config/branding/signature.txt` to the PR body.

#### 3.2 PR Body Structure

```markdown
## Summary
<1-3 bullet points describing the change>

## Changes
<Detailed list of modifications>

## Test Plan
- [ ] <Test step 1>
- [ ] <Test step 2>

---

Designed by Sebastien Rousseau — https://sebastienrousseau.com
Engineered with Euxis — Enterprise Unified Execution Intelligence System — https://euxis.co
```

#### 4.3 Version References

All documentation MUST reference the current system version from `registry.json → protocol_version`. See the Versioning & Compatibility Protocol for full rules.

#### 4.4 Documentation Sync

Run `euxis-sync-docs` to synchronize documentation with the current fleet state:
```bash

```

The `librarian` agent is responsible for documentation governance sweeps.

### 5. Commit Message Format

```
<type>(<scope>): <subject>

<body>

Engineered with Euxis — Enterprise Unified Execution Intelligence System — https://euxis.co
```

The `prepare-commit-msg` hook at `~/.euxis/cli/bin/hooks/prepare-commit-msg` appends the `Engineered with Euxis` trailer automatically.

**Type prefixes:** `feat`, `fix`, `docs`, `refactor`, `test`, `ci`, `chore`, `perf`, `security`.

### 6. Git Signing Requirements

1. `commit.gpgsign` MUST be `true` globally.
2. A signing key MUST be configured via `user.signingkey`.
3. All commits MUST be cryptographically signed.
4. `euxis-certify` Gate 5 verifies these requirements.

### 7. JSON Manifest Validation

The following manifests MUST be valid JSON:
- `registry.json`
- `squads.json`
- `capabilities.json`
- `config/codex/codex.json`

`euxis-certify` Gate 6 validates these via `jq`.

### 8. Enforcement

1. **`euxis-certify` Gate 5** — Branding compliance (signature, PR template, README attribution, hooks, signing, commit signatures, URL check, workflow, open PRs).
2. **`euxis-certify` Gate 6** — Documentation governance (sync-docs, required docs, freshness, version references, JSON manifests).
3. **`euxis-lint`** — Structural and protocol compliance checks.
4. **`gatekeeper`** — Enforces branding on all release artifacts.
5. **`reviewer`** — Verifies branding presence during quality gate reviews.

### 9. Agents with Branding Responsibilities

| Agent | Responsibility |
|-------|---------------|
| `orchestrator` | Enforces `euxis-hooks pr` usage in all delegated PR creation |
| `gatekeeper` | Branding on changelogs, release notes, and release artifacts |
| `reviewer` | Validates branding presence during quality gate |
| `librarian` | Documentation freshness and copyright header compliance |
| `writer` | Branding in user-facing documentation |
| `evangelist` | Brand voice and visual consistency |
