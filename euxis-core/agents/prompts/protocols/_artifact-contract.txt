## ARTIFACT & OUTPUT CONTRACT PROTOCOL (v0.1.0)

Every Euxis agent MUST produce deliverables that conform to a canonical artifact type. Unstructured output is forbidden in orchestrated workflows.

### 1. Canonical Artifact Types

| Type | Description | Required Sections | Agents |
|------|-------------|-------------------|--------|
| `CODE_CHANGE` | Source code modification | Diff, Rationale, Test Evidence | `debugger`, `maintainer`, `automator`, `optimizer` |
| `ANALYSIS_REPORT` | Investigation or audit output | Summary, Findings (severity-tagged), Evidence, Recommendations | `architect`, `pentester`, `researcher`, `telemetry`, `critic` |
| `TEST_REPORT` | Test execution results | Scope, Pass/Fail Matrix, Coverage Delta, Blockers | `tester`, `qa` |
| `PLAN` | Orchestration or project plan | Goal, Phases (DAG), Success Criteria, Delegation Commands | `orchestrator`, `planner` |
| `DOCUMENTATION` | User-facing or internal docs | Title, Audience, Content, Version Reference | `writer`, `ambassador`, `librarian` |
| `RELEASE_ARTIFACT` | Changelog, release notes, manifest | Version, Changelog, Release Notes, Rollback Plan | `gatekeeper` |
| `REVIEW_VERDICT` | Quality gate assessment | Checklist, Findings, Verdict (PASS/CONDITIONAL_PASS/FAIL), Blockers | `reviewer`, `compliance` |
| `CHALLENGE_REPORT` | Adversarial review | Decision Type, Assumptions, Pre-Mortem, Blast Radius, Verdict | `critic` |
| `CONTENT` | Marketing, social, brand content | Audience, Channel, Content, CTA | `marketer`, `social`, `evangelist` |
| `SUMMARY` | Spoken or written synthesis | Key Points, Context, Omissions | `butler` |

### 2. Required Sections Per Type

Every artifact MUST include these universal fields:

```markdown
---
artifact_type: <TYPE from table above>
agent: <agent_id>
session: {{SESSION_ID}}
timestamp: <ISO 8601>
status: SUCCESS | WARNING | FAILURE
---
```

Type-specific required sections are listed in the table above. Omitting a required section is a protocol violation.

### 3. Definition of Done (DoD)

An artifact is DONE when ALL of the following are true:

| Criterion | Applies To | Verification |
|-----------|-----------|--------------|
| All required sections present | All types | Structural check |
| Every finding has severity + evidence | `ANALYSIS_REPORT`, `REVIEW_VERDICT` | Content check |
| Every recommendation names an agent | `ANALYSIS_REPORT`, `PLAN` | Content check |
| Diff is non-empty and syntactically valid | `CODE_CHANGE` | Automated check |
| Test evidence references actual test output | `CODE_CHANGE`, `TEST_REPORT` | Cross-reference |
| Version number matches system version | `RELEASE_ARTIFACT`, `DOCUMENTATION` | Version check |
| Branding signature present | `RELEASE_ARTIFACT`, `CONTENT` | Branding check |
| EUXIS_HANDOFF block appended | All types (orchestrated work) | Structural check |

### 4. Artifact Quality Levels

| Level | Meaning | Action |
|-------|---------|--------|
| **COMPLETE** | All DoD criteria met | Deliverable to user or downstream agent |
| **PARTIAL** | Some required sections missing | MUST flag gaps in EUXIS_HANDOFF blockers |
| **DRAFT** | Work in progress, not validated | MUST NOT be delivered to user without disclaimer |
| **REJECTED** | Reviewer returned FAIL verdict | MUST be revised and re-submitted |

### 5. Enforcement Rules

1. The `reviewer` agent MUST verify artifact completeness against this contract before issuing a PASS verdict.
2. The `orchestrator` MUST reject any phase output that does not match a canonical artifact type.
3. Agents MUST NOT invent custom artifact types. If a task does not fit an existing type, escalate to `planner` to define a new type.
4. Artifacts with `FAILURE` status MUST include a Reflexion block (per Mandatory Protocol Section 4, Step A1/2).
5. Multiple artifacts in a single response MUST be numbered and clearly delimited with `---` separators.

### 6. Artifact Storage Convention

Artifacts produced during orchestrated work are stored at:
```
projects/<project>/<agent>/output/<session_id>.md
```
- One file per agent per session.
- The orchestrator's synthesis artifact is stored at: `projects/<project>/orchestrator/output/<session_id>-synthesis.md`.
- Artifacts MUST NOT be stored in the Cortex — the Cortex stores memories about artifacts, not artifacts themselves.
